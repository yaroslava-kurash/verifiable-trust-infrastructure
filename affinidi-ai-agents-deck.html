The problem

AI agents can act — but they have no identity.

We're handing agents our credentials, our keys, and the authority to do things on our behalf. Today that means shared secrets, static API keys, and no way to say who did what — or to cut an agent off.

🕳️

No identity

Agents borrow a human's login or a shared service account. Nothing is attributable.

🔓

Over-shared secrets

Raw API keys and passwords get pasted into prompts and configs.

🚫

No off switch

When an agent goes rogue or a laptop is lost, there's no clean way to revoke it.

The solution

A Verifiable Trust Agent + MCP

Affinidi's VTA gives each agent a cryptographic identity, a permission boundary, a secrets vault, and a signing oracle. vta-mcp exposes all of it to any MCP-speaking host — with zero custom code.

Concrete walkthrough

Stand up an agent — and revoke it

🧳 Scenario:  Nova, a personal travel agent that books flights using your airline credentials.
1

Mint its identity

The VTA mints Nova a DID from the ai-agent template.

pnm bootstrap provision-integration \ --template ai-agent --create-context
2

Scope its powers

Grant least privilege — read & release vault secrets, sign on your behalf. Nothing more.

role: application caps: vault-read, fill-release, sign-trust-task
3

Plug into the host

Point the agent host at the MCP bridge. No SDK, no glue code.

"vta": { "command": "vta-mcp", "args": ["--vta","nova"] }
4

It does the job

Nova releases the airline API key (sealed), signs the booking, proves loyalty tier with a VP.

vault_release "airline-api" sign "booking#4471" issue_vp loyalty-gold
5

See what it did

Every action is attributable to Nova's DID.

pnm audit --actor did:webvh:…:nova
6

Revoke instantly

Laptop lost or job done? One command and Nova can no longer authenticate.

pnm device wipe nova-laptop \ --reason "lost" --scope full
