Step-Up Auth: The Art of the Possible

Step-up authentication in the VTA is not a hardcoded login — it is a composable approval ceremony. Who approves, what proof they must produce, and how they are summoned are three independent dials. This is what becomes possible as we build it out with push-enabled devices and hardware authenticators.

DIDComm-first transport WebAuthn hardware gate Push-wake any device VC / VP credential approval Type-enforced verified evidence Fully audited ceremonies

The reasoning

Why design it this way

A normal session proves you logged in. Step-up proves this exact sensitive action is authorized, by the right party, with sufficient assurance, right now. Three principles make that durable.

01 — BLAST RADIUS

High-risk ops deserve more

Granting admin, revoking keys, releasing vault secrets, proxy-login — these can't ride on a 15-minute token. Step-up demands a fresh, explicit, scoped approval bound to that one operation and a single-use challenge.

02 — SEPARATION

Four-eyes, by design

Delegated modes mean a different person or device must sign off. The most dangerous actions can require a second human — turning a single compromised session into a dead end for an attacker.

03 — FLEXIBILITY

Policy, not plumbing

Raising the assurance bar — from "tap your phone" to "two officers touch hardware keys" — is a configuration change against the same core machinery. No re-architecture, no new subsystem, no bespoke crypto.

End-to-end

Worked example: quorum hardware-key approval for key/revoke

⚡

The scenario. An operator tries to revoke a production signing key. Policy requires 2-of-3 security officers to each approve from a push-woken phone by touching a FIDO security key — within a 5-minute window, or the operation is refused.

Operator VTA Push gateway Approver device Hardware key
1
● Operator → VTA

Request the sensitive operation

Operator calls key/revoke on a live, authenticated AAL1 session. The VTA's gate resolves the policy for this op-class and finds a quorum + hardware floor.

2
● VTA

Mint a single-use step-up challenge

A 256-bit challenge is bound to the session, subject, the approver pool, target assurance (aal2+), required evidence and a 5-minute TTL. The operation parks, pending approval.

403 step_up_required · acceptableEvidence: ["webauthn"]
3
● VTA → Mediator + Gateway

Summon the approvers

The VTA buffers an approve-request for each officer through its mediator and fires a push-wake to each registered gateway. The VTA never sees the APNs/FCM token — only an opaque wake handle.

auth/step-up/approve-request/0.1  ·  push/wake/0.1 (authcrypt)
4
● Gateway → Devices

Wake the pocket approvers

Each gateway fires the platform push. Officers' phones wake, connect to the mediator, drain the pending request, and render a consent screen: "Approve revocation of key X?"

5
● Device + Hardware key

Touch to authorize

Each officer confirms. The device invokes WebAuthn against the bound challenge; the FIDO key requires a physical touch + user verification before it will sign. The private key never leaves the authenticator.

approve-response/0.1 · evidence.kind=webauthn
6
● VTA

Verify evidence & count the quorum

Each assertion is verified against the challenge and resolved to a passkey in the approver's DID document. Verified evidence becomes a distinct type — a call site that forgets to verify won't compile. The VTA counts approvals until the 2-of-3 threshold is met.

7
● VTA → Operator

Elevate, execute, audit

Threshold met inside the window → the session is elevated, the parked key/revoke executes, and every step — request, wakes, each approval, the outcome — is written to the audit log. Window expires first → the operation is refused, untouched.

session → aal2 · operation committed · ceremony audited

The one-sentence pitch

Step-up auth in the VTA is a composable ceremony, not a hardcoded login. The same core machinery delivers everything from "tap to confirm on your phone" to "two security officers each touch a FIDO key from attested environments, summoned by push, within five minutes — or the operation is refused." It is standards-native throughout — DIDComm, WebAuthn, VCs/VPs, DIDs — so every future gate interoperates with external verifiers instead of being a scheme we alone must defend. And because verified evidence is enforced by the type system and every ceremony is audited, raising the bar later never risks a silently skipped check.
